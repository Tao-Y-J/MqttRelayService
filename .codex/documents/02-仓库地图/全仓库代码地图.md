## 2. 核心数据流

```
 客户端发布消息
  → MqttBrokerHost.InterceptingPublishAsync（轻量：入队 + ProcessPublish=false）
     → InMemoryMessageQueue.EnqueueAsync（有界 Channel，满载显式丢弃；异常时不回落到 Broker 默认分发）
  → MessageDeliveryService.ConsumeLoopAsync（后台循环）
    → ReadAllAsync（ChannelReader 异步挂起等待）
    → MessageRouter.RouteAsync（Topic 匹配，异常向上传播；Retain=true 时即使当前无匹配目标也继续注入 Broker）
    → IMqttBrokerHost.PublishAsync（InjectApplicationMessage 单次注入，sourceClientId 附加到 UserProperties）
    → MqttBrokerHost.InterceptingOutboundPacketAsync（出站拦截：EchoToSender=false 时阻止回发发送方）
    → 成功：状态=Succeeded / 失败：退避等待 → 重新入队 → 死信
```

## 5. 关键设计决策

### 5.1 消息注入 + 出站拦截（非客户端直投）

当前 MQTTnet 5.x 的 `InjectApplicationMessage` 是按 Topic 广播，无法严格按 ClientId 定向投递。因此采用：
- `MessageRouter.RouteAsync` 过滤匹配客户端（路由层过滤发送方）
- `MessageDeliveryService` 单次调用 `PublishAsync` 注入 Topic
- `MqttBrokerHost` 出站拦截器 `InterceptingOutboundPacketAsync` 阻止回发发送方（Broker 分发层二次保险）
- 注入消息通过 MQTT 5.0 `UserProperties` 附加 `x-source-client-id` 标记

### 5.2 多消费者并发

`MessageDeliveryService.StartAsync` 根据 `ReliabilityOptions.MaxConcurrentHandlers` 启动对应数量的消费者任务（最小值为 1）。每个消费者独立从 `ReadAllAsync` 读取消息并处理。

运行期转发失败后，通过 `ScheduleRetryEnqueueAsync` 将退避延迟和重新入队调度到后台任务执行，当前消费者立即返回处理后续消息，不会被退避延迟阻塞。后台重试调度任务受 `ReliabilityOptions.MaxPendingRetryTasks` 限制，超限消息直接进入死信，避免失败洪峰绕过有界队列造成无界内存增长。停机 drain 阶段则使用 `DelayAndRequeueDuringStopAsync` 同步等待退避结束后再重新入队，以保证 drain 不会在消息重新出现之前提前结束。

Retained Message 的 `Retain` 标志从 `MqttBrokerHost.OnInterceptingPublishAsync` 写入 `RouteContext`，由 `MessageDeliveryService` 传给 `IMqttBrokerHost.PublishAsync` 并在 Broker 注入时恢复。`Retain=true` 且当前没有匹配订阅者时，投递服务仍会注入 Broker，让 MQTTnet保存运行期内存 retained 消息；服务进程重启后 retained 消息不会恢复。

### 5.3 停机排空

`MessageDeliveryService.StopAsync` 两阶段执行：
1. 取消 `_cts`，消费者循环的 `ReadAllAsync` 退出
2. 等待所有消费者任务完成
3. 单线程 drain：使用 `TryDequeueAsync` 非阻塞循环消费队列剩余消息，直到超时或队列为空
4. 记录 `DrainedCount` 和 `RemainingCount`

Drain 阶段若因超时取消触发 `OperationCanceledException`，停止 drain 并记录剩余数量。

### 5.4 队列满载策略

统一使用 `BoundedChannelFullMode.Wait`，避免 `DropWrite` 导致 `TryWrite` 返回 `true` 但消息被静默丢弃。

`DropWhenQueueFull=true` 时：应用层显式检查 `Count >= Capacity`，满则返回 `false` 并记录 Warning。

### 5.5 客户端发布拦截失败语义

`MqttBrokerHost.OnInterceptingPublishAsync` 对客户端原始发布先设置 `ProcessPublish=false`，再执行活动时间更新、Payload 转换和入队。拦截链路发生异常时，消息不会回落到 Broker 默认分发路径，避免绕过内部队列、重试、死信和 `EchoToSender` 控制。

服务端注入的转发消息通过 `x-relay-message-id` 识别，识别成功后保留 Broker 默认分发路径。

### 5.6 客户端会话快照

`ClientRegistry` 内部仍使用 `ConcurrentDictionary` 管理在线客户端，写入和更新订阅时只修改注册表内部对象；`GetSessionAsync` 和 `GetAllSessionsAsync` 返回会话快照，路由线程枚举的是独立 `HashSet`，不会与订阅事件并发修改同一个集合。

`MqttBrokerHost.OnClientConnectedAsync` 会为每次连接生成 `ConnectionId` 并写入 MQTTnet `SessionItems`，`OnClientDisconnectedAsync` 注销时带回该值。`ClientRegistry.UnregisterAsync` 只移除与当前 `ConnectionId` 匹配的会话，旧连接的延迟断开事件不会误删同 `ClientId` 的新连接。

### 5.7 BrokerWorker 启停边界

`BrokerWorker` 初次启动 Broker 失败时调用 `IHostApplicationLifetime.StopApplication()`，避免后台服务异常被 Host 配置忽略后进程假存活。停机时先调用 `base.StopAsync` 取消监控循环，再停止 Broker，避免停机期间监控循环看到 `IsRunning=false` 后触发重启。

### 5.8 统一 Kestrel 单端口与自适应静态托管设计

- **统一 Kestrel 单端口监听**：在主服务注册拦截装饰器收集指标后，通过单个 Web 端口同时对外提供轻量级 API 端点与静态大屏，避免 5000 / 5001 重复暴露同一套站点。
- **自适应物理路径回退**：Dashboard 支持物理静态网页托管（自适应 `AppContext.BaseDirectory/wwwroot` 与开发期 `ContentRootPath/wwwroot`），完美兼容了本地 `dotnet run` 一键调试与 Windows 服务生产运行部署。
- **无代理直连极致性能**：大屏直接通过相对路径 `/api/` 访问主服务的指标和审计日志接口，无需通过 HttpClient 代理，彻底避免了跨域 (CORS) 限制与后台代理代理损耗。
