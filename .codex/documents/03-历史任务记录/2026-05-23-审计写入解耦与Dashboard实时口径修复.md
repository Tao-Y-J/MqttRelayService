# 审计写入解耦与 Dashboard 实时口径修复

## 1. 本次实际变更

- `MetricsService` 不再为每条消息审计事件直接 `Task.Run` 一个独立落库任务。
- `MetricsService` 新增单后台写入循环，按 `MessageId` 合并待写审计状态后顺序写库。
- Dashboard 首页 `/api/metrics` 恢复为读取进程内实时计数器与实时日志窗口，不再优先改读审计库汇总。

## 2. 这样修改的原因

- 原实现会在 MQTT 热路径上为每条消息额外创建一个后台任务，而 `AuditRepository` 内部又用单写锁串行落库，高并发压测下会快速积压大量待写任务。
- Dashboard 首页又优先读取审计库汇总，因此即使压测脚本已经停止，延迟写入的审计任务仍会继续推动首页指标变化，形成“停了以后数字还在追”的表现。
- 现在把消息审计写入收敛为单后台循环，并按同一 `MessageId` 做最新状态覆盖，减少了无效写库次数，也把写库延迟与首页实时指标解耦。

## 3. 当前边界

- Dashboard 首页展示的是当前进程内的实时运行视图，强调低延迟和低干扰。
- 审计分页接口仍然读取持久化数据库，保留历史查询能力。
- 消息审计表仍然只保存每条消息的最新状态，不扩展为一条消息多阶段全历史流水。
