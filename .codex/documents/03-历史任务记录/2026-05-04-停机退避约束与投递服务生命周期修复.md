# 历史任务记录：停机退避约束与投递服务生命周期修复

## 1. 任务信息

- **日期**：2026-05-04
- **目标**：修复停机 drain 期间的退避配置边界提示，并清理 `MessageDeliveryService` 的可复用生命周期状态

## 2. 实际实现

- `MessageDeliveryService.StartAsync` 新增启动态保护；服务已启动时记录 Warning 并忽略重复启动请求。
- `MessageDeliveryService.StartAsync` 新增可靠性配置告警：当 `ShutdownDrainTimeoutMs < RetryMaxDelayMs` 时记录 Warning，明确停机 drain 可能早于单次最大退避结束。
- `MessageDeliveryService.StopAsync` 在 `finally` 中统一执行生命周期清理：释放 `_cts`、置空 `_cts`、清空 `_consumerTasks`、清理已完成的 `_retryTasks`、恢复 `_isStopping=false`。
- `ReliabilityOptions.ShutdownDrainTimeoutMs` 注释补充实际约束：若希望停机阶段至少覆盖一次失败消息的最大退避等待，该值必须不小于 `RetryMaxDelayMs`。
- `README.md` 增补停机排空超时与最大退避时间的关系说明。

## 3. 验证

- `MessageDeliveryServiceTests` 新增“启动告警”和“Stop 后再次 Start”两条回归测试。
- 执行 `dotnet test --no-restore`，66 个测试全部通过。

## 4. 边界说明

- 当前实现继续使用内存队列；停机阶段若保留回队列失败，消息仍按现有逻辑进入死信。
- 停机 drain 是否能等待完单次最大退避，直接取决于 `ShutdownDrainTimeoutMs` 与 `RetryMaxDelayMs` 的配置关系。
