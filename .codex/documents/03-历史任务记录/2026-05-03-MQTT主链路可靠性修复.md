## 关键决策

### 1. 单次 Topic 注入（非客户端直投）

当前 MQTTnet 5.x 的 `InjectApplicationMessage` 是按 Topic 广播，无法严格按 ClientId 定向投递。因此采用最小修复：
- `MessageRouter.RouteAsync` 仍负责判断匹配目标并过滤发送方
- `MessageDeliveryService` 不再对每个 target 循环注入，改为单次注入
- 日志记录目标数量，但不声称逐个客户端定向投递
- 如果未来需要严格按客户端投递，需另开任务研究 MQTTnet 5 定向投递能力

### 2. 消费循环改为 ChannelReader.ReadAllAsync

扩展 `IMessageQueue` 接口，新增 `ReadAllAsync` 返回 `IAsyncEnumerable<ForwardMessage>`，由 `ChannelReader.ReadAllAsync` 直接实现。消费循环改用 `await foreach`，空队列时真正异步挂起，取消 token 后正确退出，无需固定间隔轮询。

### 3. 重试退避实现方式

在 `HandleFailureAsync` 中计算退避延迟后，先 `await Task.Delay(delay)` 再重新入队。当前 `MaxConcurrentHandlers=1`，阻塞影响有限，实现最小且逻辑直观。如果后续性能要求提高，再开任务实现独立延迟重试队列。

### 4. EchoToSender=false 真实运行语义

MQTT 协议本身不包含发送方标识，Broker 分发时无法识别原始发布者。因此采用应用层方案：
- `MessageDeliveryService` 调用 `PublishAsync` 时传入 `sourceClientId`
- `MqttBrokerHost.PublishAsync` 将 `sourceClientId` 附加到消息的 `UserProperties`（`x-source-client-id`）
- `MqttBrokerHost` 注册 `InterceptingOutboundPacketAsync` 出站拦截器
- 当检测到出站数据包是 `MqttPublishPacket`、包含 `x-source-client-id`、且目标 `ClientId` 等于该值时，设置 `ProcessPacket = false` 阻止分发
- 该方案依赖 MQTT 5.0 的 User Properties 特性，且只在 `EchoToSender=false` 时生效

### 5. 停机队列排空

`StopAsync` 分两个阶段执行：阶段 1 取消 `_cts` 让 `ReadAllAsync` 退出消费循环；阶段 2 使用 `TryDequeueAsync`（非阻塞）循环处理队列剩余消息，直到 `ShutdownDrainTimeoutMs` 超时或队列为空。最后记录排空数量和剩余数量。

### 6. 队列满载策略

- 不再使用 `BoundedChannelFullMode.DropWrite`（根因：TryWrite 在丢弃时返回 true，调用方无法感知）
- 统一使用 `BoundedChannelFullMode.Wait`
- DropWhenQueueFull=true 时：入队前显式检查 Count >= Capacity，如果是则记录 Warning 日志并返回 false
- DropWhenQueueFull=false 时：WriteAsync + EnqueueTimeoutMs 超时返回 false + Warning 日志

## 可靠性边界声明

本次修复后，以下边界仍然成立：

1. **投递语义**：当前版本按 Topic 注入，由 Broker 分发给匹配订阅者，不提供严格客户端级直投
2. **队列类型**：内存队列（InMemoryMessageQueue），进程退出时未处理消息丢失
3. **消费者并发**：`MaxConcurrentHandlers` 配置已生效，默认 1，可配置为更多消费者并行处理
4. **重试退避**：`HandleFailureAsync` 中使用 `Task.Delay(delay)` 阻塞当前消费者，其他消费者继续处理新消息。未实现独立延迟队列
5. **至少一次**：消息在 MQTT QoS 重发或应用层重试场景下会重复投递，消费端需具备幂等处理能力
6. **磁盘队列**：未实现，可靠性边界限于进程存活期间

## 未实现边界

1. **性能测试**：本轮未执行高并发吞吐和延迟压测
2. **定向投递**：未实现严格按 ClientId 的定向消息注入，当前采用 Topic 注入 + 出站拦截方案
