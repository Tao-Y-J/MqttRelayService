## 关键决策

### 1. 单次 Topic 注入（非客户端直投）

当前 MQTTnet 5.x 的 `InjectApplicationMessage` 是按 Topic 广播，无法严格按 ClientId 定向投递。因此采用最小修复：
- `MessageRouter.RouteAsync` 仍负责判断匹配目标并过滤发送方
- `MessageDeliveryService` 不再对每个 target 循环注入，改为单次注入
- 日志记录目标数量，但不声称逐个客户端定向投递
- 如果未来需要严格按客户端投递，需另开任务研究 MQTTnet 5 定向投递能力

### 2. 消费循环改为 ChannelReader.ReadAllAsync

替代方案对比：
- **方案 A**（采用）：扩展 IMessageQueue 接口，新增 `ReadAllAsync` 返回 `IAsyncEnumerable<ForwardMessage>`，由 `ChannelReader.ReadAllAsync` 实现
  - 优点：语义清晰，与 .NET Channel 生态一致，取消传播正确
  - 缺点：需要修改接口
- **方案 B**（放弃）：在 ConsumeLoopAsync 中使用 `WaitToReadAsync` + `TryRead`
  - 优点：不改接口
  - 缺点：代码更冗长，语义不如 `await foreach` 清晰

### 3. 重试退避实现方式

替代方案对比：
- **方案 A**（采用）：在 HandleFailureAsync 中 `await Task.Delay(delay)` 后重新入队
  - 优点：最小修改，逻辑直观
  - 缺点：单线程消费者会被阻塞（MaxConcurrentHandlers=1）
- **方案 B**（放弃）：实现独立延迟重试队列/优先级队列
  - 优点：消费者不被阻塞，吞吐量更高
  - 缺点：改动量大，需要新组件和复杂调度逻辑

**决策**：当前 MaxConcurrentHandlers=1，阻塞影响有限。如果后续性能要求提高，再开任务实现独立延迟队列。

### 4. 队列满载策略

- 不再使用 `BoundedChannelFullMode.DropWrite`（根因：TryWrite 在丢弃时返回 true，调用方无法感知）
- 统一使用 `BoundedChannelFullMode.Wait`
- DropWhenQueueFull=true 时：入队前显式检查 Count >= Capacity，满载直接返回 false + Warning 日志
- DropWhenQueueFull=false 时：WriteAsync + EnqueueTimeoutMs 超时返回 false + Warning 日志

## 可靠性边界声明

本次修复后，以下边界仍然成立：

1. **投递语义**：当前版本按 Topic 注入，由 Broker 分发给匹配订阅者，不提供严格客户端级直投
2. **队列类型**：内存队列（InMemoryMessageQueue），进程退出时未处理消息丢失
3. **消费者并发**：单消费者（MaxConcurrentHandlers=1），重试退避会阻塞消费者
4. **至少一次**：消息可能被重复投递（MQTT 协议层面 + 重试机制），消费者需做好幂等处理
5. **磁盘队列**：未实现，可靠性边界限于进程存活期间
