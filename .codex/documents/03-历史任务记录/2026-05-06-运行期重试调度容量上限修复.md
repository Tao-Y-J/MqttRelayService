# 运行期重试调度容量上限修复

## 目标

修复运行期非阻塞重试调度绕过内部有界队列容量的问题，确保 Broker 注入持续失败时，内存中的待退避重试任务数量有硬上限。

## 实际修改

- `ReliabilityOptions` 新增 `MaxPendingRetryTasks`，默认值为 `1000`。
- `MessageDeliveryService` 在创建后台重试调度任务前检查当前待重试任务数量。
- 当前待重试任务数量达到 `MaxPendingRetryTasks` 时，新失败消息直接写入死信。
- 已完成的后台重试调度任务会从跟踪集合移除，长期运行时不会堆积已完成任务引用。
- `appsettings.json` 和 `appsettings.Development.json` 写入 `MaxPendingRetryTasks` 默认配置。
- `MessageDeliveryServiceTests` 增加超出后台重试调度上限时写入死信的单元测试。

## 边界

- 内部转发队列仍由 `QueueCapacity` 约束。
- 运行期等待退避的后台重试调度任务由 `MaxPendingRetryTasks` 约束。
- 超出后台重试调度上限的消息不再继续等待退避，直接进入死信。
- 停机 drain 阶段仍使用同步退避等待和重新入队逻辑，不创建后台重试调度任务。

## 验证

- 执行 `dotnet test MqttRelayService.slnx --no-restore`。
- 执行 `git diff --check`。
