# Web 管理面板实现说明

本文档记录了 Web 监控管理面板（Dashboard）的架构设计、技术实现和运行边界，作为后续维护的长期上下文事实。

## 1. 目标与运行机制

### 1.1 目标
- 构建一个极具现代感、响应式且完全解耦的 Web 监控管理面板。
- 允许内网用户无需身份验证（免密公开访问）直接查看 MQTT Broker 运行状态、吞吐速率、队列水位、客户端连接、消息日志和死信明细。
- **融合多端口设计**：Dashboard 静态大屏与指标 API 直接集成并融合于主服务 `MqttRelayService` 中，共享同一个应用程序进程与请求处理管道。通过一次部署，即可利用 Kestrel 同时监听多端口独立提供服务。
- **无侵入拦截**：对主项目现有的核心业务逻辑（`InMemoryMessageQueue`、`MqttBrokerHost`、`DeadLetterService`）达到 **100% 零修改**，全部统计指标均通过 DI 容器的装饰器进行优雅拦截。

### 1.2 拓扑架构
- **统一多端口宿主端 (MqttRelayService)**：
  - **端口 5000 (API 端口)**：使用极简的轻量 Kestrel API 提供纯 JSON 格式的指标、分页审计和控制 Minimal APIs（`/api/metrics`、`/api/messages`、`/api/settings/throughput` 等）。
  - **端口 5001 (大屏端口)**：使用 static files 托管 `wwwroot` 下的前端玻璃拟态（Glassmorphic UI）高大上单页面应用（SPA），并通过 SPA Fallback 响应默认根路径。大屏页面直接通过相对路径 `/api/` 访问主服务的接口，完全避免了浏览器跨域限制，且实现了零网络代理损耗。

## 2. 关键技术决策

### 2.1 零侵入指标拦截（装饰器模式）
我们没有修改任何原有业务逻辑文件，而是直接在 DI 容器中注入了装饰器拦截事件：
- `MetricsMessageQueue` 包装并代理 `IMessageQueue`：拦截 `EnqueueAsync`，实时统计成功入队数和队列饱合度。
- `MetricsMqttBrokerHost` 包装并代理 `IMqttBrokerHost`：拦截 `PublishAsync`，计算转发成功率与分发响应耗时（LatencyMs）。
- `MetricsDeadLetterService` 包装并代理 `IDeadLetterService`：拦截 `WriteAsync`，捕获死信日志及导致失败的异常原因。

### 2.2 线程安全高性能统计指标服务（MetricsService）
`MetricsService` 作为单例服务注册，包含：
- **原子计数器**：使用 `Interlocked` 维护全局收到、转发成功、转发失败、重试次数和死信的原子增量计数。
- **滚动日志队列**：使用 `ConcurrentQueue` 维护最近 100 条消息状态的轻量快照，供日志详情页实时枚举。
- **历史吞吐采样环形缓冲区**：后台配置每 2 秒一次的采样任务，通过增量计算获取收到速率、转发速率和死信速率。环形缓冲区保留最近 60 条记录（共 2 分钟历史），用于动态波形图的绘制。
- **大消息 Payload 滑动内存缓存**：使用 `ConcurrentDictionary` 缓存最近 100 条消息的实际 Payload（单条消息大小上限 8KB），供前端点击具体消息流进行详细报文审计。内存上限严格有界，完美防范内存溢出。

### 2.3 相对路径直连避开 CORS
由于大屏与主服务 API 整合在同一个 ASP.NET Core 应用中：
- 静态页面在大屏端口 (5001) 加载后，通过相对路径直接请求 `/api/metrics` 等接口，浏览器发出的请求同样会落入 5001 端口的统一 Kestrel 请求管道进行路由解析，这不仅物理上彻底避免了跨域 (CORS) 限制，也消除了跨进程 `HttpClient` 代理调用的代理损耗和网络抖动，获得极致的响应速度。

### 2.4 静态托管路径双向自适应回退

为了兼顾本地开发期与 Windows 服务生产运行期的部署差异，Dashboard 静态文件托管采用了一套 **自适应物理路径回退机制** 与 **MSBuild 属性复制** 结合的方案：
1. **静态托管双路径回退**：在 `Program.cs` 启动时，首选定位 `AppContext.BaseDirectory/wwwroot` 目录。若该目录不存在（如直接通过 `dotnet run` 调试运行且尚未执行发布或打包复制），则自动回退到 `app.Environment.ContentRootPath/wwwroot`（即项目源码所在的物理目录），完美兼容了本地一键调试体验与 Windows 服务（当前目录会被切换为 `C:\Windows\System32`）的运行路径限制。
2. **MSBuild 自动复制输出**：在 `MqttRelayService.csproj` 中，配置 `<Content Update="wwwroot\**\*">` 并在构建期设置 `CopyToOutputDirectory` 为 `PreserveNewest`。这保证了在项目构建（Build）或发布（Publish）时，前端 `wwwroot` 静态资源目录会自愈且高可靠地输出到发布 bin 目录中。

### 2.5 内存消息日志的就地可变状态更新（仅保留最终状态）

- **技术决策与目的**：
  在早期的内存日志设计中，每一次对消息接收（Received/Queued）、转发（Forwarded）和死信（DeadLetter）的记录都会将一个新的匿名对象推入滑动窗口队列，这在面板消息流列表中会导致同一条消息产生重复、分离的状态更新行（例如先有一行 "Queued" 状态，随后又产生一行 "Succeeded" 状态）。
  为了解决这一问题并完美遵循“不需要跟踪同一条消息的各种状态，只需要跟踪最终状态，状态是可变的”设计原则，我们重构了 `MetricsService` 中 `_messageLogs` 的数据管理结构。
- **具体实现细节**：
  1. **双数据结构滑动缓存**：使用 `ConcurrentDictionary<string, MessageLogEntry>` 按 `MessageId` 唯一索引管理日志记录，并配合 `ConcurrentQueue<string>` 滑动队列管理其生命周期，防止内存无界增长。
  2. **就地 Upsert 覆盖**：新增强类型 `MessageLogEntry` 结构。在 `RecordReceived`、`RecordForwarded` 和 `RecordDeadLetter` 被调用时，对内存中该 `MessageId` 对应的记录执行就地原子更新（`AddOrUpdate`），更新其当前 `Status`、`LatencyMs`、`RetryCount`、`Timestamp` 和 `ErrorMessage`，仅保存并呈现其最终最新状态。
  3. **生存期驱逐机制**：当插入新消息导致字典大小超过 `MaxLogCount` (100 条) 时，在滑动队列中移出最旧的消息 ID 并从字典中原子擦除，保持极佳的定长内存开销与物理安全边界。
  4. **时间戳高效排序**：API 端拉取数据时，通过 `SystemTimestamp` 倒序对所有条目进行高效内存排序，保证前端展示最新活动消息在最上方。

### 2.6 解决转发成功依然残留入队堆积状态 (AsyncLocal 跨调用上下文透传 MessageId)

- **技术决策与问题定位**：
  在原始设计中，由于 `IMqttBrokerHost.PublishAsync(...)` 接口签名未包含 `MessageId` 参数，投递服务 `MessageDeliveryService` 转发注入消息时，外围拦截装饰器 `MetricsMqttBrokerHost` 被迫在此接口处调用 `Guid.NewGuid().ToString("N")` 生成新 ID 写入 `RecordForwarded`。因此，即使消息在入队接收阶段生成了 ID 并被标记为 "Queued"，在转发阶段指标层收到的却是另一个随机新 ID，从而引发大屏物理审计中残留多条无法对应更新的 "Queued" 消息的 Bug。
- **具体实现细节**：
  1. **设计无侵入数据通道**：引入静态上下文 `MessageContext.CurrentMessageId`，内部包装 `AsyncLocal<string?>` 变量，该变量可以跨异步调用栈上下文自动沿调用流单向传递（Thread-safe & Task-isolated）。
  2. **原位 ID 复用**：在 `MessageDeliveryService.TryForwardAsync` 中，调用转发注入之前将当前消息的原始 `MessageId` 注入到 `MessageContext.CurrentMessageId.Value` 中，并在 `finally` 块中原子清理归 `null`。
  3. **装饰器感知复用**：`MetricsMqttBrokerHost.PublishAsync` 在拦截时，检查该 `AsyncLocal` 的值，若存在则复用其值作为 `RouteContext.MessageId` 调用 `RecordForwarded`，若为空则平滑退化至生成新 Guid，确保在第三方直接调用 `PublishAsync` 时依然具备高鲁棒性。
  4. **就地状态原子覆盖**：由于转发成功时所传递的 `MessageId` 与入队时相同，`MetricsService` 的 `AddOrUpdateLog` 和 `AuditRepository` 将在底层执行就地原子覆盖更新。大屏在不需要刷新页面的情况下，消息的“入队堆积”状态将在审计日志中被原位替换为“转发成功”，在大屏仅显示一条唯一 MessageId 的最新状态记录，无任何冗余多余条目。

## 3. 运行边界与性能保障
- **微秒级拦截回调**：所有装饰器拦截动作仅在内存中执行简单的 `Interlocked` 计数或轻量入队，不执行任何 IO 阻塞操作，绝对不会影响 MQTT Broker 的高并发吞吐性能。
- **定长内存限制**：滚动日志窗口固定为 100 条，吞吐历史记录固定为 60 条，Payload 滑动缓存上限 100 条（限制单条最大 8KB）。此限制不仅保证了内存占用极低且恒定，更避免了无界内存增长的风险。

## 4. 验证与交付情况
- **单元与集成测试**：本地 `dotnet test` 执行共计 86 个测试用例，100% 成功通过，包括原先的主链路测试及新增的 Dashboard 拦截器与代理测试，无任何退化。
- **功能性验证**：已在本地启动 `MqttRelayService`，同时监听 1883 (MQTT)、5000 (API) 和 5001 (Dashboard) 端口。高大上磨砂大屏 ApexCharts 图表以 1.5 秒频率平滑更新，客户端订阅树和历史日志正常拉取展示，点击最近消息的“查看”按钮可高亮美化解析 Payload JSON。

## 5. 本地开发一键启停脚本

为了方便本地开发人员一键启动与优雅关闭主服务和 Web Dashboard，我们实现了两个专属开发批处理脚本：

### 5.1 一键启动开发脚本 (start-dev.cmd)

- **存放路径**：[start-dev.cmd](file:///d:/SourceCodes/MqttRelayService/start-dev.cmd)
- **文件编码**：**纯 ASCII（0-127）** 编码。通过全面剔除所有中文字符与 BOM 字节，完全排除了 Windows 不同语言区域设置及 CMD 代码页切换对批处理解析的干扰，实现跨语系 Windows 环境下的 100% 绝对稳定运行。
- **运行机理**：
  1. 进程自愈：先执行强力端口与进程清理，保证没有历史残留程序霸占端口。
  2. 极简并发：使用 `start` 异步指令在独立控制台窗口运行 `src\MqttRelayService` 主服务项目，一次性拉起并托管 1883 (MQTT)、5000 (API)、5001 (Dashboard) 端口，简化开发观察与维护复杂度。

### 5.2 一键关闭开发脚本 (stop-dev.cmd)

- **存放路径**：[stop-dev.cmd](file:///d:/SourceCodes/MqttRelayService/stop-dev.cmd)
- **文件编码**：**纯 ASCII（0-127）** 编码，完全免于乱码与字节拆分报错。
- **运行机理**：
  1. 进程树强杀：通过 `taskkill /f /t /im MqttRelayService.exe` 精准杀灭子运行期实例。整棵进程树被杀后，宿主 `dotnet.exe` 会立刻优雅消亡，从而在不误杀其他 dotnet 进程的前提下 100% 释放 1883、5000、5001 端口。
  2. 桌面窗口自清理：使用 `taskkill /f /fi "WINDOWTITLE eq MqttRelayService*"` 匹配并回收由 `start-dev.cmd` 创建的两个独立控制台，还原干净整洁的开发桌面。
