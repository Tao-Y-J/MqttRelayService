# 2026-05-06 Review 问题修复：启动失败、脚本迁移与停机顺序测试

## 1. 变更目标

本次修复处理最近一次 review 提出的 3 个问题：

1. `MessageDeliveryService` 在检测到残留消费者或残留重试任务时，会拒绝重启但仍返回成功，导致 Host 和 Windows Service 假存活。
2. Windows Service 安装/卸载脚本只按当前 `Service.Name` 查找服务，部署后改名会遗留旧服务。
3. `HostShutdownOrderTests` 没有绑定 `Program.cs` 的真实后台服务注册顺序，无法防止注册顺序回归。

## 2. 实际修改

### 2.1 投递服务启动失败改为显式异常

- 文件：`src/MqttRelayService/Services/Implementations/MessageDeliveryService.cs`
- 做法：
  - 启动前发现残留重试任务时，改为记录错误并抛出 `InvalidOperationException`。
  - 启动前发现残留消费者时，改为记录错误并抛出 `InvalidOperationException`。
  - 新增 `ThrowStartupBlocked` 方法统一处理日志与异常。
- 结果：
  - `DeliveryWorker` 启动过程中不会再把“拒绝重启”误判为成功启动。
  - Host 会明确收到启动失败信号，不再出现投递链路未启动但进程仍显示 Running 的状态。

### 2.2 后台服务注册顺序提取为可测试入口

- 文件：`src/MqttRelayService/Program.cs`
- 做法：
  - 新增 `Program.RegisterHostedServices(IServiceCollection services)`。
  - `Main` 中改为调用该方法完成后台服务注册。
- 结果：
  - 后台服务的真实注册顺序可以被测试直接验证。
  - `QueueMetricsWorker -> DeliveryWorker -> BrokerWorker` 的停机安全顺序被固定到代码入口。

### 2.3 Windows Service 脚本兼容服务改名迁移

- 文件：
  - `src/MqttRelayService/Scripts/install-service.ps1`
  - `src/MqttRelayService/Scripts/uninstall-service.ps1`
- 做法：
  - 安装脚本新增“按当前部署目录可执行路径探测服务”逻辑。
  - 安装前会卸载：
    - 当前 `Service.Name`
    - 默认服务名 `MqttRelayService`
    - 与当前 `MqttRelayService.exe` 路径匹配的已安装服务
  - 卸载脚本新增 `ServiceNames` 参数，支持安装脚本传入多个候选服务名。
  - 未显式传入参数时，卸载脚本也会按配置名称、默认名称和当前部署路径自动匹配。
- 结果：
  - 同一部署目录下即使修改过 `Service.Name`，重新安装或卸载也不会遗留旧服务。

## 3. 测试更新

- 文件：
  - `tests/MqttRelayService.Tests/DeliveryWorkerTests.cs`
  - `tests/MqttRelayService.Tests/HostShutdownOrderTests.cs`

- 做法：
  - `DeliveryWorkerTests` 新增启动失败向上传播测试。

  - `MessageDeliveryServiceTests`：

    - 新增“残留重试调度拒绝启动”测试。
