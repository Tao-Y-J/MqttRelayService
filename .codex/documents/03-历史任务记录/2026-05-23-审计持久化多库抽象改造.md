# 审计持久化多库抽象改造

## 1. 实际改动

- 将 `ISqliteAuditRepository` / `SqliteAuditRepository` 重构为 `IAuditRepository` / `AuditRepository`，公开代码不再暴露 SQLite 专有命名。
- 新增 `AuditStorageOptions`，通过 `AuditStorage` 配置中的 `Provider` 字符串直接映射 `SqlSugar DbType`，决定底层数据库提供程序与连接字符串。
- 审计仓储统一改为 `SqlSugar` API：建表走 `CodeFirst`，写入走 `Insertable` / `Updateable`，查询与清理走 `Queryable` / `Deleteable`，仓库代码中不再保留手写 SQL。
- Dashboard 文案与 API 注释统一改为“审计持久化”语义；本地开发默认配置放在 `appsettings.Development.json`，主配置面向正式数据库。

## 2. 结果边界

- 本地开发默认仍可使用 SQLite，但该事实只保留在 Development 配置，不再作为仓库公开架构概念。
- 正式环境需要显式提供 `AuditStorage:Provider` 与 `AuditStorage:ConnectionString`，启动时按照配置初始化审计表结构。
- 当前保留策略仍是基于条数的自动清理，消息与客户端历史的保留上限由 `AuditStorage` 配置控制。
