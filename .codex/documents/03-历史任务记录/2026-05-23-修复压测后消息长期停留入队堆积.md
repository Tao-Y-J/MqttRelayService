# 修复压测后消息长期停留入队堆积

## 1. 实际修改

本次修复处理的是 Web 管理面板中“消息审计列表长期残留 `Queued`/入队堆积”的状态悬挂问题。

实际实现如下：

- `MetricsService.RecordReceived` 仅在消息首次进入系统时写入 `Queued` 审计初始态。
- 对重试回队、停机保留回队这类同一消息再次入队场景，仍更新内存日志，但不再把持久化审计状态无条件重写回 `Queued`。
- `MetricsService` 的待落库缓存对同一 `MessageId` 增加状态优先级合并规则：
  - `DeadLetter` 最高。
  - `Succeeded` 与 `Failed` 为终态，高于 `Forwarding`、`Routing`、`Queued`。
  - 当缓存中已经存在终态时，后续到达的中间态不会覆盖终态。
- `Dashboard` 总览计数在启用审计仓储时改为读取审计汇总：
  - `TotalReceived` 取审计总记录数。
  - `TotalPending` 统计 `Queued`、`Routing`、`Forwarding`。
  - `TotalSucceeded`、`TotalFailed`、`TotalDeadLetter` 按最终状态聚合。
- `未命中订阅者` 继续保留为 `IsSubscriberHit = false` 的完成结果，不再按错误风格展示，前端命中文案调整为“已转发未命中”。

## 2. 结果边界

- Broker 注入成功仍然定义为系统处理完成，即使未命中订阅者，也不会再被视为堆积。
- 真正的未决消息只包括当前仍处于 `Queued`、`Routing`、`Forwarding` 的记录。
- 本次没有修改消息转发主链路、重试策略或死信策略，只修正审计状态收敛与总览统计口径。
