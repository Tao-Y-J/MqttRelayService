# SQLite 审计追平速度优化

## 1. 实际修改

- `AuditRepository.RecordMessageAuditsAsync(...)` 不再对每条消息执行一次 `AnyAsync(MessageId)` 再分别 `Insert/Update`。
- 新实现先对整批 `MessageId` 做一次集合查询，得到已存在主键集合后，再分成：
  - `toInsert`
  - `toUpdate`
  两个批次执行。
- `RecordMessageAuditAsync(...)` 也统一复用同一套批量 Upsert 内部路径，避免单条写入和批量写入逻辑分叉。
- `MetricsService` 的后台审计刷盘批次从 `100` 提高到 `1000`，减少高并发压测收尾阶段的数据库往返次数和事务次数。

## 2. 为什么这样改

- 之前的实现虽然表面上叫“批量写入”，但实际仍是：
  - 单全局写锁串行
  - 每条记录先查一次是否存在
  - 再逐条 `Insert` 或 `Update`
- 在几十万消息压测后，这种模式会导致：
  - 连接客户端已经全部退出
  - 内存队列已经排空
  - 但 SQLite 仍然要花很长时间慢慢补写审计
- 这次优化的目标就是把“压测结束后的审计追平时间”从分钟级压到秒级。

## 3. 当前边界

- 本次没有修改 UI，也没有修改消息转发、重试、死信策略。
- 本次优化的是 SQLite 审计持久化吞吐，不是把 Dashboard 首页重新切回内存口径。
- 若后续仍需继续压缩追平时间，下一步应优先考虑：
  - 更高效的 ORM/原生批量 Upsert
  - 读写分离或更轻量的审计落盘结构
