# Review 新增问题修复：停机与重启边界

## 1. 目标

修复仓库 Review 新识别出的两个运行时问题：Broker 连续重启失败后的服务假存活，以及投递服务停机时在途消息丢失。

## 2. 实际修改

- `BrokerWorker` 新增监控间隔、重启间隔和最大连续失败次数字段；Broker 启动后如果连续重启失败次数超过上限，立即调用 `IHostApplicationLifetime.StopApplication()` 请求 Host 停止，不再仅退出监控循环。
- `MessageDeliveryService.ConsumeLoopAsync` 对停机取消导致的 `OperationCanceledException` 单独处理；消费者在退出前先尝试把已取出但尚未完成的在途消息重新入队。
- `MessageDeliveryService` 新增 `PreserveInFlightMessageAsync`；重新入队成功时交由 `StopAsync` 的 drain 阶段继续处理，重新入队失败时将消息转入死信。
- `BrokerWorkerTests` 新增连续重启失败耗尽后请求 Host 停止的回归测试。
- `MessageDeliveryServiceTests` 新增停机取消在途消息后重新入队并由 drain 阶段处理完成的回归测试。

## 3. 关键决策

- Broker 连续重启失败达到上限后直接停止 Host，因为此时服务已经失去 MQTT 监听能力，继续保持 Windows Service 的 Running 状态会误导运维和监控。
- 停机取消中的在途消息先保回内部队列，因为本服务的可靠性边界是“队列 + 后台投递 + drain”；消费者取消本身不应直接把已取出的消息丢掉。
- 保回队列失败时转入死信，因为此时消息已经脱离原始队列，必须留下可追踪的失败记录。
