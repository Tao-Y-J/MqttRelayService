# 仓库 Review 问题修复

## 1. 目标

修复仓库 Review 中确认的四类运行时风险：客户端发布拦截异常后的默认分发回退、Broker 初次启动失败后的服务假存活、客户端订阅集合并发读写、BrokerWorker 停机期间监控循环重启竞态。

## 2. 实际修改

- `MqttBrokerHost.OnInterceptingPublishAsync` 对客户端原始发布先设置 `ProcessPublish=false`，再执行活动时间更新、Payload 转换和内部队列入队。拦截链路失败时，消息不再回落到 Broker 默认分发路径。
- `ClientRegistry` 注册时复制外部传入会话，读取会话时返回快照，订阅更新和活动时间更新在内部会话对象上加锁，路由线程枚举独立快照集合。
- `BrokerWorker` 构造函数新增 `IHostApplicationLifetime`，初次启动 Broker 失败时调用 `StopApplication()`，避免服务进程继续假存活。
- `BrokerWorker.StopAsync` 调整为先执行 `base.StopAsync` 取消监控循环，再调用 `IMqttBrokerHost.StopAsync` 停止 Broker，停机期间不会被监控循环重新拉起。
- 新增 `BrokerWorkerTests` 覆盖启动失败请求 Host 停止。
- 新增集成测试覆盖发布拦截异常时订阅者不会收到绕过内部队列的默认分发消息。
- 新增注册表并发测试覆盖订阅更新与路由并发执行不抛异常。

## 3. 关键决策

- 发布拦截失败按阻断处理，因为当前服务的可靠性链路以内部队列、后台投递、重试和死信为边界，默认分发会绕过这些边界。
- 客户端会话采用快照读取，因为路由只需要某一时刻的在线客户端和订阅视图，不需要持有注册表内部可变集合。
- Broker 初次启动失败直接请求 Host 停止，因为此时服务没有 MQTT 监听能力，继续运行会误导部署和监控。
- 停机顺序以先停止监控循环为准，因为 Broker 停止会让 `IsRunning=false`，监控循环未退出时会误判为异常停止。

## 4. 验证

- 执行 `dotnet test MqttRelayService.slnx`。
- 结果：62 个测试全部通过。

## 5. 边界

- 本次未引入磁盘队列或持久化。
- 本次未改变现有 Topic 注入 + 出站拦截的转发模型。
- 本次未改变认证配置和 Windows Service 脚本。
