# 修复消息转发延迟统计为 0.0 ms 的问题

## 1. 本次实际修改

- **Models / MessageContext**：扩展 `MessageContext` 模型，新增两个基于 `AsyncLocal` 的字段 `CurrentFirstReceivedAt` 和 `CurrentRetryCount`，提供轻量且零侵入的异步链路上下文数据传递机制。
- **Services / MessageDeliveryService**：
  - 修改私有方法 `TryForwardAsync` 的签名，改为直接接收整个 `ForwardMessage` 对象而不是仅 `RouteContext`。
  - 在 `TryForwardAsync` 中，调用 `_brokerHost.PublishAsync` 之前，将消息的接收时间 `context.Timestamp` 和 `message.RetryCount` 写入 `MessageContext`。
  - 在 `finally` 块中，主动将上述两个上下文属性置为 `null`，确保绝对不会发生内存泄漏或异步调用链路交叉污染。
- **Services / MetricsMqttBrokerHost (装饰器)**：
  - 拦截 `PublishAsync` 时，优先从 `MessageContext` 中获取原始消息的 `firstReceivedAt`（首次接收时间）和 `retryCount`（已重试次数）。
  - 若 `firstReceivedAt` 存在，则计算当前的端到端总延迟：`(DateTime.Now - firstReceivedAt.Value).TotalMilliseconds`，并作为 `latencyMs` 传入 `_metrics.RecordForwarded`；同时 `context.Timestamp` 保持对齐首次接收时间。
  - 若 `firstReceivedAt` 不存在（回退兼容外部/测试直连模式），则使用原有的 `Stopwatch` 注入物理耗时。
  - 转发指标写入时，调用 `_metrics.RecordForwarded` 传入真实的已重试次数，不再始终写死为 0。
- **Tests / MessageDeliveryServiceTests**：
  - 更新私有反射测试用例 `TryForwardAsync_CancelledToken_RethrowsOperationCanceledException`，将其传参从 `RouteContext` 修正为 `ForwardMessage`，使单元测试 100% 顺畅通过。

## 2. 边界条件与技术决策

- **端到端投递耗时**：投递延迟真实对齐了从“外部客户端发布成功入队”到“内部成功转发注入 Broker 并触发回调”的完整生存期开销，即使在有高并发排队积压或多次退避重试时，大屏上也能显示精确累加的真实延迟（例如 `5235.1 ms`），而不是原先纯内存操作时的 `0.0 ms`。
- **完全兼容与回退**：若有外部组件（如集成测试）未通过投递服务直接调用 BrokerHost 的 `PublishAsync`，装饰器检测到 `firstReceivedAt` 为 `null` 时会平滑回退，仅测量注入的 Stopwatch 物理耗时，保障全局运行高可用。
