# SQLite 审计最终态追平优化

## 1. 实际修改

- `MetricsService` 的后台审计 writer 增加 50ms 合并窗口，先让同一 `MessageId` 的 Queued 与终态在内存待写区合并，再批量写入数据库。
- `MetricsService.MergeAuditRecord(...)` 在终态优先保留时同步刷新 Topic、SourceClientId、PayloadSize、QoS、Retain 等展示字段，确保最终态快照仍携带最新消息元数据。
- `AuditRepository` 继续使用 SqlSugar ORM 路径，不写 SQLite 专用 SQL；SQLite 当前批量查询和批量写入块从保守小批次放大到 2000 / 500，减少压测收尾阶段的 ORM 往返次数。
- `message_audit` 仍是单表最终态快照，每个 `MessageId` 只保留一条最新状态记录，不保存状态事件流水。

## 2. 技术边界

- 本次没有修改 MQTT 转发、重试、死信、客户端注册或 Web API 路由语义。
- SQLite 仍是当前临时运行库，仓储实现保持 SqlSugar ORM 兼容，后续切换服务器数据库时继续复用同一仓储抽象。
- Dashboard 和 `/api/messages` 继续展示每条消息当前最新态/最终态，Payload 回查仍从内存缓存或 `message_audit` 快照读取。

## 3. 验证

- 新增仓储测试覆盖同批重复 `MessageId` 只保留最新态、批量 Queued 更新为 Succeeded 后不丢行，以及更新时不覆盖首次 `CreatedAt`。
- 新增指标测试覆盖 Queued 与 Succeeded 在同一刷盘窗口内只写一次最终态。
